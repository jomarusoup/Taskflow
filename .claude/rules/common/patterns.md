## 불변성 (매우 중요)
항상 새로운 객체를 생성하고, 기존 객체를 절대 변경하지 마십시오.

// Pseudocode
WRONG:  modify(original, field, value) → changes original in-place
CORRECT: update(original, field, value) → returns new copy with change
근거: 불변 데이터는 숨겨진 부작용을 방지하고, 디버깅을 용이하게 하며, 안전한 동시성을 가능하게 합니다.

## 파일 정리

작은 파일 여러 개 > 큰 파일 몇 개:

- 높은 응집력, 낮은 결합도
- 일반적으로 200~400줄, 최대 800줄
- 대형 모듈에서 유틸리티를 추출합니다.
- 유형별이 아닌 기능/도메인별로 정리하세요.

## 오류 처리

오류는 항상 종합적으로 처리하십시오.

- 모든 단계에서 오류를 명시적으로 처리하세요.
- 사용자 인터페이스에 표시되는 코드에 사용자가 이해하기 쉬운 오류 메시지를 제공하세요.
- 서버 측에서 오류에 대한 자세한 컨텍스트를 로그에 기록합니다.
- 실수를 절대 묵묵히 받아들이지 마세요.

## 입력 유효성 검사
시스템 경계에서 항상 유효성을 검사하십시오.

- 처리하기 전에 모든 사용자 입력을 검증하십시오.
- 가능한 경우 스키마 기반 유효성 검사를 사용하십시오.
- 명확한 오류 메시지를 통해 빠르게 실패를 알릴 수 있습니다.
- 외부 데이터(API 응답, 사용자 입력, 파일 내용)는 절대 신뢰하지 마십시오.

## 코드 품질 체크리스트

과제 완료를 표시하기 전에:

- [ ] 코드는 읽기 쉽고 이름도 잘 지정되어 있습니다.
- [ ] 함수의 크기가 작습니다(<50줄).
- [ ] 파일은 핵심 내용만 담고 있습니다(<800줄).
- [ ] 중첩 수준이 4단계를 초과하지 않아야 합니다.
- [ ] 적절한 오류 처리
- [ ] 하드코딩된 값은 사용하지 마세요 (상수 또는 설정 파일을 사용하세요).
- [ ] 변이 없음 (불변 패턴 사용)
